# Source-to-Segment Map (`/build-source-map`)

An auto-generated reverse index that maps source file paths to the business logic segments that reference them. Answers "which source files are documented where?" and "which source files have NO segment coverage?" in one glance.

## The file

Located at `.business-logic/<repo>/SOURCE_MAP.md`.

Auto-generated by `scripts/build-source-map.sh`. **Never edit manually.**

Format:

```markdown
# Source-to-Segment Map

Auto-generated by build-source-map.sh. Do not edit manually.

| Source File | Segment(s) |
|-------------|------------|
| `features/Email/EmailRules.ts` | 01-smtp-config.md |
| `models/Email/EmailSchema.ts` | 06-infra-schema.md |
| `services/ProposalCommissionService.ts` | 04-commission-engine.md, 03-proposal-creation.md |
| `globals/Aws.ts` | _(no segment)_ |
| ... | ... |
```

## Command

### `/build-source-map`

**Workflow:**
1. Read all segment files from `.business-logic/<repo>/split/`
2. Extract every `Code:` entry: `- <file>:<line> (construct)`
3. Build inverse map: file → [segment1, segment2, ...]
4. Write `SOURCE_MAP.md`
5. Report stats: "N source files mapped, M have no segment coverage"

The agent extracts file paths from the Code: entries already in every segment. It normalizes paths (strips `source/` prefix, handles relative paths). If a source file appears in the tree but has zero entries in the map, it's flagged as "no segment coverage."

## What it enables

| Use case | Before | After |
|----------|--------|-------|
| Find undocumented files | Manual grep through all segments | Look for `_(no segment)_` entries |
| Impact analysis | "Which segments reference this file?" | Single row lookup |
| Staleness check | Manual cross-ref of git diff vs segments | "file X changed → segments A,B may be stale" |
| PR review scope | "Which rules are affected?" | Immediate segment list from changed files |
| Gap analysis | Scan everything | Only scan files with `_(no segment)_` |

## Integration with Staleness Check

The staleness check (`/check-staleness`) uses SOURCE_MAP.md as its primary input:
1. Get changed files from manifest diff
2. Look up each changed file in SOURCE_MAP.md
3. Flag the segments that reference changed files as ⚠️ stale

## Integration with Gap Agent

The Gap Agent (`/gap-scan`) uses SOURCE_MAP.md to optimize:
1. Read SOURCE_MAP.md
2. Find files with `_(no segment)_` — these are high-priority targets
3. Find files recently changed (from manifest diff) — these are medium priority
4. Skip files that are unchanged and already mapped
5. Scan only the priority set

## Token impact

SOURCE_MAP.md is loaded **only during `/build-source-map`, `/check-staleness`, and `/gap-scan`**. It is NOT loaded during normal sessions. Estimated budget: ~400t for a repo with 200 source files.
